However, the bulk of the changes are **not routine updates** — they represent a **systematic one-position shift in camera-site-to-camera-name mappings** across a large block of consecutively numbered camera sites (roughly cam279–cam312). In effect, each camera site in that range has been reassigned the name, coordinates, and metadata of its neighboring site. **Significance: HIGH.** If applied as-is, the app would display the wrong camera name and location for ~20+ cameras, breaking the user experience and producing misleading map pins.

### 🔴 Port Format Change Across All Shifted Records

All shifted cameras also show a port change from a full RTSP URL (e.g., `rtsp://158.59.15.203:8003/live/cam282.stream`) to just a port number (`8003`). This is a **format regression** — the local dataset stores complete RTSP URLs while the fresh data provides only port numbers. Notably, the old port URLs reference the *previous* cam site number (e.g., cam282 stores `.../cam281.stream`), which is consistent with the off-by-one pattern in the existing data. This needs careful investigation to understand whether the URL construction logic in the app can handle bare port numbers.
